# Feature Specification: 브라우저를 통한 다운로드 파일 전달

**Feature Branch**: `002-browser-file-download`

**Created**: 2026-06-16

**Status**: Draft

**Input**: User description: "유튜브에서 다운받은 파일을 브라우저를 통해 로컬 PC에 다운로드 가능하도록 구현해줘"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 완료된 다운로드 파일을 브라우저로 받기 (Priority: P1)

서버에서 유튜브 영상 다운로드 작업이 완료되면, 사용자는 브라우저의 다운로드 기능을 통해 해당 파일을 자신의 로컬 PC로 저장할 수 있다.

**Why this priority**: 현재 시스템은 서버 로컬 스토리지에만 파일을 저장하고 사용자에게 전달할 방법이 없다(001 기능의 범위 제외 사항). 이 기능이 없으면 사용자는 다운로드 결과물을 실제로 받을 수 없으므로, 전체 서비스의 가치가 완성되지 않는다.

**Independent Test**: 다운로드 작업을 완료 상태로 만든 뒤, 완료 화면에서 "다운로드" 동작을 실행하면 브라우저가 파일을 사용자의 기본 다운로드 폴더(또는 사용자가 선택한 위치)에 저장하는지 확인하면 독립적으로 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 다운로드 작업이 완료 상태일 때, **When** 사용자가 완료된 항목에서 다운로드 동작을 실행하면, **Then** 브라우저가 해당 파일 전송을 시작하고 사용자의 로컬 PC에 파일이 저장된다.
2. **Given** 파일이 mp4 형식으로 저장되어 있을 때, **When** 사용자가 다운로드하면, **Then** 받은 파일의 확장자와 내용이 mp4 영상과 일치한다.
3. **Given** 파일이 mp3 형식으로 저장되어 있을 때, **When** 사용자가 다운로드하면, **Then** 받은 파일의 확장자와 내용이 mp3 음원과 일치한다.
4. **Given** 다운로드 작업이 아직 완료되지 않았을 때, **When** 사용자가 다운로드를 시도하면, **Then** 시스템은 다운로드를 거부하고 아직 준비되지 않았음을 알린다.

---

### User Story 2 - 원본 영상 제목 기반 파일명 (Priority: P2)

사용자가 파일을 받을 때, 저장되는 파일명이 원본 유튜브 영상 제목을 기반으로 하여 사용자가 내용을 쉽게 식별할 수 있다.

**Why this priority**: 핵심 전달 기능(User Story 1) 없이는 의미가 없지만, 파일명이 식별 불가능한 값(예: 임시 ID)이면 사용자 경험이 크게 저하된다.

**Independent Test**: 다운로드를 실행했을 때 브라우저가 제안하는 파일명이 영상 제목(파일 시스템에서 허용되지 않는 문자는 안전하게 치환됨)과 확장자를 포함하는지 확인.

**Acceptance Scenarios**:

1. **Given** 영상 제목이 "테스트 영상!"이고 형식이 mp4일 때, **When** 사용자가 파일을 다운로드하면, **Then** 저장되는 파일명은 파일 시스템에서 허용되는 문자로 정리된 제목과 ".mp4" 확장자를 포함한다.
2. **Given** 두 개의 다른 작업이 동일한 영상 제목을 가질 때, **When** 각각 다운로드하면, **Then** 두 파일은 덮어쓰지 않고 구분 가능한 파일명으로 저장된다.

---

### User Story 3 - 다운로드 중 전송 진행 표시 (Priority: P3)

파일 크기가 크거나 네트워크가 느린 경우, 사용자는 브라우저로의 파일 전송이 진행 중임을 알 수 있다.

**Why this priority**: 브라우저 자체의 다운로드 표시 기능으로 상당 부분 충족되며, 핵심 가치(파일 전달)에는 영향을 주지 않는 보조 기능이다.

**Independent Test**: 큰 파일을 다운로드할 때 전송이 끊기지 않고 완료되며, 사용자가 전송 중임을 인지할 수 있는 표시(브라우저 기본 또는 앱 내)가 있는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 큰 파일을 다운로드하기 시작했을 때, **When** 전송이 진행되는 동안, **Then** 전송이 중간에 끊기지 않고 끝까지 완료된다.
2. **Given** 전송이 완료되었을 때, **When** 사용자가 파일을 확인하면, **Then** 파일 크기가 서버에 저장된 원본 파일 크기와 일치한다(손상 없음).

---

### Edge Cases

- 사용자가 이미 삭제되었거나 만료된(보관 기간 초과) 파일을 다운로드 시도하면 어떻게 처리하는가?
- 동일한 파일을 여러 번 반복해서 다운로드 요청하면 어떻게 처리하는가?
- 다운로드 도중 네트워크가 끊겨 전송이 중단되면 어떻게 처리하는가? (이어받기 지원 여부)
- 사용자가 다른 사용자의 다운로드 작업 결과물에 접근을 시도하면 어떻게 처리하는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 완료 상태의 다운로드 작업에 대해 사용자가 브라우저를 통해 파일을 받을 수 있는 동작(예: 다운로드 버튼/링크)을 제공해야 한다 (MUST).
- **FR-002**: 시스템은 완료되지 않은(대기중/진행중/실패) 작업에 대한 파일 전달 요청을 거부하고 명확한 안내 메시지를 제공해야 한다 (MUST).
- **FR-003**: 시스템은 전달하는 파일의 형식(mp4/mp3)에 맞는 콘텐츠 형식을 사용자의 브라우저에 알려, 브라우저가 올바르게 처리(저장)하도록 해야 한다 (MUST).
- **FR-004**: 시스템은 사용자에게 전달되는 파일의 이름을 원본 영상 제목 기반으로 지정해야 하며, 파일 시스템에서 허용되지 않는 문자는 안전한 문자로 치환해야 한다 (MUST).
- **FR-005**: 시스템은 동일한 파일명이 충돌할 경우에도 각 다운로드 작업의 결과 파일을 구분하여 전달해야 한다 (MUST).
- **FR-006**: 시스템은 전달 과정에서 파일이 손상되거나 잘리지 않고 원본과 동일하게 전송되도록 해야 한다 (MUST).
- **FR-007**: 시스템은 사용자가 본인이 요청한 다운로드 작업의 결과물에만 접근할 수 있도록 제한해야 한다 (MUST).
- **FR-008**: 시스템은 삭제되었거나 만료된 작업의 파일 요청에 대해 명확한 오류 메시지를 표시해야 한다 (MUST).
- **FR-009**: 시스템은 대용량 파일 전송 시에도 전체 파일이 끝까지 안정적으로 전달되도록 해야 한다 (MUST).

### Key Entities

- **다운로드 작업 (DownloadJob)**: 001 기능에서 정의된 기존 엔티티. 본 기능에서는 완료 상태의 작업이 가진 저장 파일 경로/식별자를 사용하여 사용자에게 파일을 전달하는 데 사용된다.
- **파일 전달 요청 (FileDeliveryRequest)**: 사용자가 특정 다운로드 작업의 결과 파일을 받기 위해 보내는 요청. 속성: 대상 작업 식별자, 요청 시각, 처리 결과(성공/거부/오류).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 다운로드가 완료된 작업의 100%에 대해 사용자가 브라우저를 통해 파일을 받을 수 있다.
- **SC-002**: 사용자가 다운로드 버튼을 클릭한 후 1초 이내에 브라우저의 파일 전송이 시작된다.
- **SC-003**: 전달된 파일은 100% 원본과 동일한 내용(크기/무결성)을 가진다.
- **SC-004**: 완료되지 않았거나 존재하지 않는 작업에 대한 다운로드 시도 시 100% 명확한 오류 안내가 제공된다.
- **SC-005**: 사용자는 받은 파일의 제목만으로 어떤 영상인지 추가 확인 없이 식별할 수 있다.

## Assumptions

- 001 기능(유튜브 동영상 다운로더)에서 서버는 이미 다운로드된 파일을 로컬 스토리지에 보관하고 있으며, 본 기능은 그 파일을 사용자 브라우저로 전달하는 단계만 다룬다.
- 파일 보관 기간 및 자동 삭제 정책은 별도 운영 정책으로 정해지며, 본 기능은 "만료/삭제된 파일 요청 시 오류 처리"만 다루고 보관 기간 자체의 수치는 범위에서 제외한다.
- 인증/계정 시스템이 별도로 존재한다는 전제 없이, "본인이 요청한 작업"은 현재 세션 또는 작업 식별자 소유 여부로 판별한다고 가정한다.
- 이어받기(resume) 다운로드는 브라우저 표준 동작에 위임하며, 별도의 서버측 이어받기 로직 구현은 본 기능 범위에서 제외한다.
